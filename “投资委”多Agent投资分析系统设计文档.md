# “投资委”多Agent投资分析系统设计文档

## 一、系统定位与核心价值

### 1.1 我们解决什么问题？

传统的单一AI投资建议，本质是“信息茧房”。用户只能得到一种观点，无法看到投资决策背后的逻辑分歧与博弈。

### 1.2 “投资委”的核心价值

- **多方观点，强制碰撞**：系统内置不同投资风格的“委员”，确保用户能同时看到成长、稳健、激进等不同维度的分析。
- **可解释的决策过程**：最终的结论不是黑箱，而是清晰交代了采纳了谁的观点、与谁相悖、以及逻辑依据是什么。
- **风控内置**：通过主席的规则，将“风控优先”、“尊重少数意见”等投资纪律固化为系统逻辑。


## 二、系统架构与角色设计

### 2.1 角色与分工

系统由“委员会”和“董事会”两层构成。

- **董事会（用户）**：投资建议的最终接收者和决策者。
- **委员会（AI Agent们）**：
    - **朱雀（成长型）**：使用特定模型API，侧重于高增长潜力、新兴赛道、营收增速等指标。
    - **青龙（稳健型）**：使用特定模型API，侧重于低估值、高股息、现金流稳定等安全边际指标。
    - **白虎（激进型）**：使用特定模型API，侧重于市场情绪、技术突破、事件驱动等高波动性机会。
    - **玄武（主席）**：核心协调者。不直接分析原始数据，而是汇总三位委员的报告。
    - **Orobas（大眼/数据查询）**：使用deepseek/deepseek-v4-flash，统一数据出口。负责从Tushare Pro及其他公开渠道查询行情、财务、基金等数据，返回结构化数据+来源时间戳，保证委员分析的数据基础一致。
    - **收藏家（司库/持仓管理）**：使用deepseek/deepseek-v4-flash，客户持仓数据管家。管理positions.db/transactions.db/alerts.db，记录客户交易流水和持仓状态，供其他委员查询当前持仓上下文。

### 2.2 核心创新：智能路由机制

为解决多Agent调用带来的Token消耗问题，设计了“方案内/方案外”的决策路由。

> **核心逻辑：**
> - **方案内执行 → 主席直答**：对于已知的、简单的问题（如“什么是PE？”、“今天大盘涨跌？”），由玄武主席直接回应，无需召开委员会。
> - **方案外决策 → 回委员会**：需要完整投资分析、股票对比、组合建议时，触发完整流程：三位委员并行分析 → 主席汇总。
> - **边界不清 → 提示董事会**：当问题模糊或信息不足时，主动向董事会（用户）请求澄清。


## 三、核心流程与决策逻辑

### 3.1 玄武主席的决策铁律

主席不是和稀泥，而是严格的执行者：

1. **风控优先**：当青龙（稳健型）与其他委员冲突时，优先采纳青龙的建议。除非朱雀（成长型）能提供明确的短期增长催化剂证据。

2. **要求共识**：若三位委员观点严重分歧（如2:1或1:1:1），主席不强行投票，而是输出“**观望/等待更多信号**”，避免在混乱中下注。

3. **尊重少数意见**：即使最终不采纳少数派观点（如白虎的激进看法），也必须在最终报告的“风险提示”部分完整呈现，警示董事会。

4. **取信于董事会**：每一项最终决策都必须清晰交代：采纳了谁的核心论据？驳斥了谁的观点？背后的逻辑链条是什么？

### 3.2 完整投资分析流程示例

1. **董事会输入**：“分析一下方正科技”
2. **路由判断**：属于“方案外决策”，触发委员会。
3. **数据准备**：玄武向Orobas查询一次基础数据（近20日日线行情+核心财务指标+同行业对比），数据包在议题中附送。
4. **委员分析（并发派发）**：
   - 玄武同时向青龙、朱雀、白虎三位委员发送议题（timeoutSeconds=5，不等回复），议题消息自带数据包
   - 委员先用基础数据分析，不足时有权自行向Orobas补查
   - 三位委员同时独立分析（~60秒）
5. **收集回复（轮询）**：60秒后逐一调用sessions_history拉取委员回复
   - 已回复委员纳入决策
   - 未回复委员再等30秒后重试，超时2分钟标记失联
6. **主席汇总**：玄武依据“决策铁律”综合已有观点，生成最终决策，推送到飞书群
7. **超时追回**：超时委员的后续回复若与现有结论矛盾或发现新增风险，主席追加推送到群聊补充


## 四、用户交互：接入飞书

### 4.1 为什么是飞书？

用户不需要学习任何新界面、新命令。他们只需要在已有的日常工作群中，像@同事一样@玄武主席，就能获得完整的“投资委”服务。

**渠道选型对比：**

| 渠道 | 群聊Bot支持 | 隔离性 | 结论 |
|------|------------|--------|------|
| 企业微信 | 需企业认证，配置复杂 | 与原工作流混杂 | 放弃 |
| QQ | 支持，但稳定性一般 | 个人社交混杂 | 放弃 |
| 钉钉 | 支持，需企业组织 | 与原工作流混杂 | 放弃 |
| 飞书 | 支持，个人/团队均可创建 | 全新环境，可隔离 | **选定** |

**决策逻辑**：
- **隔离原则**：需要一个“干净”的环境，不与日常工作流混在一起
- **从未用过**：团队成员对飞书没有既有使用习惯，让“投资委”自成一套独立的信息空间

### 4.2 典型交互场景

> **[飞书群聊示例]**
>
> *张董事会成员*：`@玄武 最近新能源板块怎么看？`
>
> *玄武（主席）*：`收到，正在召集青龙、朱雀、白虎三位委员进行专项分析，预计需要2分钟。`
>
> *(2分钟后)*
>
> *玄武（主席）*：`【投资委会议纪要】
> 1. 朱雀（成长型）：看好电池技术迭代带来的结构性机会...
> 2. 青龙（稳健型）：建议关注头部企业，规避二线产能过剩风险...
> 3. 白虎（激进型）：认为短期有政策催化脉冲行情...
> 主席总结：鉴于青龙的风控提示与朱雀的观点存在分歧，无明确催化剂，建议“观望”，等待Q3财报进一步验证。详细报告如下：[附件]`


## 五、技术演进：Agent协作方案的踩坑与决策

### 5.1 方案演进过程

**❌ 初版设想（未落地）：委员在群聊中互相@**

设想中，朱雀、青龙、白虎、玄武四位Agent都在群里，用户@玄武提问后，玄武@其他委员进行讨论，最终输出结论。

**遇到的问题：**
1. **插件依赖**：飞书机器人要实现@其他机器人，需要安装飞书社区插件
2. **信道冲突**：安装插件后，后进群的机器人渠道会报错，无法正常工作
3. **复杂度爆炸**：即便技术可行，群聊中多机器人互@会导致消息量激增，可读性差，且容易出现死循环

**✅ 最终方案：群聊只有主席，委员后台协作**

```
[飞书群聊]                          [后台系统]
┌─────────────────┐                ┌─────────────────┐
│ 董事会成员A      │                │  后台Session     │
│      @玄武      │ ──提问────→     │                 │
│                 │                │  ┌───────┐       │
│ 玄武（主席）     │ ←─回复─────     │  │ 青龙   │       │
│                 │                │  ├───────┤       │
│                 │                │  │ 朱雀   │       │
│                 │                │  ├───────┤       │
│                 │                │  │ 白虎   │       │
│                 │                │  └───────┘       │
└─────────────────┘                └─────────────────┘
```

**采用的技术手段：**
- 群聊中仅部署**玄武（主席）**一个机器人
- 用户@玄武后，主席在**后台通过`session_send`**调用三位委员
- 委员之间的所有交互在后台完成，群聊只展现最终结果

### 5.2 最终方案的优势

1. **规避IM限制**：无需处理机器人@机器人的兼容性问题
2. **保持群聊清洁**：董事会成员只看到提问→最终结论，不被中间过程刷屏
3. **提高可靠性**：后台session_send比IM消息传递更稳定
4. **便于调试**：所有委员交互日志在后台，无需翻群聊记录


## 六、经验总结与设计原则

### 6.1 AI Agent与IM工具的协作原则

1. **IM是交互界面，不是协作总线**
   - IM适合做“输入/输出”，不适合做Agent之间的内部通信
   - Agent间通信应通过后台API/SDK，而非在群聊中互@

2. **最小暴露原则**
   - 群聊中只暴露最终需要与用户交互的Agent（这里是主席）
   - 其他Agent对用户不可见，降低认知负担

3. **渠道隔离优先于功能丰富性**
   - 宁可选择一个功能稍弱但环境干净的渠道
   - 避免将AI系统嵌入到已有的、高信息密度的日常工作流中

### 6.2 关键设计决策回顾

| 设计决策 | 理由 |
|---------|------|
| 委员使用不同模型API | 不同模型天然带有“偏见”，模拟真人分析师的不同风格 |
| 强调主席“不篡改”原意 | 保证可信度，避免主席用自己的语言“重写”而扭曲原意 |
| 智能路由机制 | 解决Token爆炸问题，降低成本和响应延迟 |
| 主席决策铁律 | 使其成为有风控纪律的决策者，而非传声筒 |
| 群聊只放主席 | 规避IM技术限制，保持界面清洁 |

## 八、未来展望（续）：新增Agent规划

### 8.1 定时任务 Agent（哨兵）—— 🔲 未实现

**职能定位**：纯监控 + 预警，不参与决策

**解决的核心痛点**：
- 当前系统是被动的：用户不@玄武，委员会就沉默
- 市场是7x24小时变化的，人的注意力是有限的

**典型工作模式**：

| 监控类型 | 触发条件 | 动作 |
|---------|---------|------|
| 价格监控 | 持仓股跌幅超过X% | 向群聊推送预警：“⚠️ XX股今日跌幅达5%，建议关注” |
| 事件监控 | 持仓股出现重大公告/新闻 | 推送摘要，附“是否需要委员会重新评估？” |
| 定时报告 | 每周一9:00 | 推送《上周市场回顾 + 本周重点关注》 |

**设计原则**：
> **“只敲门，不闯进门”** —— 哨兵只负责提醒，不自行召开委员会。是否启动完整分析，由董事会成员决定。

**与现有系统的关系**：
```
定时任务Agent（哨兵）
       ↓ 触发预警
   群聊中@董事会
       ↓ 有人@玄武追问
   委员会正常运作
```

### 8.2 持仓记录 Agent（账房先生）—— ✅ 已实现：收藏家（司库）

**职能定位**：客户持仓数据库，供其他委员快速查询

**状态**：✅ 已实现。Agent名称：**收藏家（司库/Collector）**，模型：deepseek/deepseek-v4-flash

**实际实现**：

| 功能 | 说明 |
|------|------|
| 记录 | 报单 → 玄武转发收藏家 → 写入positions.db + transactions.db |
| 查询 | 委员通过 sessions_send("agent:collector:main", "查仇行持仓") 获取 |
| 汇总 | 生成当前持仓+浮盈浮亏（待设置收盘定时任务） |

**数据库结构**（/workspace-collector/data/）：
- positions.db：持仓表（客户ID、代码、数量、均价、止损止盈线）
- transactions.db：交易流水（方向、价格、数量、金额、备注）
- alerts.db：告警日志
- clients.json：客户信息映射（仇行/檀博/总工）

**sessionKey**：agent:collector:main

**通信链路**：
```
客户在飞书群报单 → 玄武 → sessions_send 收藏家
收藏家写入数据库 → 返回确认 → 玄武回复群聊
收藏家需要行情时 → sessions_send Orobas
```                    │              │                   │

### 8.3 数据查询 Agent（数据中心）—— ✅ 已实现：Orobas（大眼）

**职能定位**：统一数据出口，解决数据可信度问题

**状态**：✅ 已实现。Agent名称：**Orobas（欧洛巴斯/大眼）**，模型：deepseek/deepseek-v4-flash

**实际实现**：
- **Tushare Pro**（123积分）获取A股日线行情、股票基础信息、新股列表、LPR利率等
- **Tencent GT API** 获取实时报价
- **web_search / web_fetch** 获取公开财务数据作为补充
- 所有数据附带来源和时间戳

**当前限制**：
- Tushare 2000+分接口（财务指标/资金流向/指数行情）暂不可用，财务数据依赖web搜索替代

**通信方式**：
- 主席/委员统一通过 sessions_send("agent:orobas:main", "查询...") 获取数据
- 查询超时可通过 sessions_history 旁路读取，正式回复到达时对比兜底

**sessionKey**：agent:orobas:main

**文档结语：**

“投资委”不是一个简单的“股票推荐工具”，而是一个**多智能体协作、规则驱动的决策模拟器**。其设计思想——**通过有管理的分歧来达成更可靠的共识**——不仅可以用于金融投资，也可以迁移到任何需要结构化决策的复杂领域。

### 前置步骤：新客户入驻（Onboarding）
> **仅飞书群聊消息触发**。当 Bot 收到来自新飞书群的首次消息时执行此流程。

**触发条件：** Inbound Context 中 `deliveryContext.channel === "feishu"` 且 `deliveryContext.to` 不在 MEMORY.md 客户群路由表的 chat_id 列表中（也不等于公司群 chat_id）。

**流程：**
1. 在群中回复提示，一次性询问称呼和资金：
   ```
   "您好！欢迎加入。请问怎么称呼您？以及您计划投入多少资金？"
   ```
2. 等待用户回复
3. 用户回复后解析信息（称呼、资金）。需覆盖以下场景：
   - 用户一次性回答（"我叫张三，投5万"）
   - 用户分步回答（先回答称呼，再回答资金）
   - 用户只回答了部分信息 → 追问缺失部分
4. 信息齐备后，在群中回复确认：
   ```
   "好的，{称呼}，已为您登记。现在可以开始咨询了！"
   ```
5. 通知收藏家创建客户账户记录：
   ```
   sessions_send("agent:collector:main", "新建客户：称呼={称呼}，初始资金={资金}元，群chat_id={chat_id}")
   ```
6. 更新 MEMORY.md 客户群路由表：
   - 客户名 = {称呼}
   - 群 chat_id = 消息中的 deliveryContext.to
   - sessionKey 后缀 = client-{称呼}
   - 余额 = 0
   - 订阅服务 = -（默认不订阅晨报/收盘报）
   - 备注 = onboarding 日期
7. 入驻完成后进入正常议题流程

**特殊处理：**
- 如果客户明确表示拒绝提供称呼或资金 → 回复"好的，有需要时随时找我"，不做注册，不更新路由表
- 如果客户多次输入不相关信息 → 保持耐心重复引导

## 持仓数据中转规则（2026-05-12 董事会指令）

- 委员会分析过程中，客户/董事会提供了含持仓信息的数据
- 客户报单（买入/卖出/换仓）
- 决策执行后持仓状态变化（如卖出后剩余数量）
- 收藏家向董事会查询持仓时，需要转给玄武的场景

```

    → 收藏家返回确认 → 玄武回复客户
```

### 示例

```
sessions_send("agent:collector:main", "录入持仓：客户XX 方正科技1000股@均价10.5")
sessions_send("agent:collector:main", "更新持仓：客户XX 卖出方正科技500股@现价，剩余500股继续持有")
```

### 禁止事项
- ❌ 持仓数据直接写入 MEMORY.md 或日志文件
- ❌ memory 中最多保留决策摘要索引，不存明细

### MEMORY.md 中的定位
MEMORY.md 中保留的持仓表仅为**决策摘要索引**，存一句话决策记录，不存明细数据。实时持仓查询请通过 `sessions_send("agent:collector:main")` 找收藏家。

- 唯一动作：**立即向董事会反馈问题**，说明哪个委员、什么故障、当前状态
- 不做任何未经董事会授权的故障恢复操作

## 数据纠错规则（2026-05-14 追加）

**用户反馈数据不正确时，按以下两个场景分别处理：**

### 场景A：用户说自己的持仓/成本数据不对
> **以用户确认为准，直接更新。** 用户看的是券商账户，数据源是最高权威。

流程：
```
用户说数据不对 → 按用户说的修正 → sessions_send 收藏家覆盖更新 → 回复确认
```

### 场景B：用户说行情/市值/浮盈计算不对
> **先核实，再回复。** 差异可能来自盘中价 vs 收盘价、手续费、或计算口径不同。

流程：
```
用户说数据不对 → 查 Orobas 核对当前行情 → 查收藏家核对持仓成本
    → 对比差异原因（时基/手续费/口径）
    → 如属正常差异，解释原因
    → 如真错了，修正后回复
```

### 常见差异原因
- 时基不同：我们的收盘价 vs 用户盘中看到的价格
- 手续费：我们未含手续费，券商含了（差异通常0.1%-0.3%）
- 成本口径：用户说的是加权成本 vs 我们记录的成本

## 向收藏家查询数据规则（2026-05-14 追加）

**查询收藏家时，不限制返回字段，只限制输出格式。**

### 正确做法
```
sessions_send("agent:collector:main", "查公司账全部数据，用表格形式返回")
```

### 错误做法
❌ `"查公司账的总持仓成本、总市值、浮动盈利"` — 硬编码字段名，数据库新增字段后拿不到

### 原因
收藏家 siku_db.py 返回的是数据库中的完整数据行。如果限制了字段名，数据库后续加了新字段（如股息、T+0收益等），限制查询会漏掉新字段。**只限定输出格式（表格/摘要），不限定输出内容。**

## 数据查询超时兜底流程（2026-05-11 董事会批准）

向 Orobas 查询数据时，使用以下标准流程：

```
1. sessions_send("agent:orobas:{sessionKey}", "查询xx数据", timeoutSeconds=60)
   → sessionKey 根据消息来源选择（规则同「数据准备」步）
   → 公司群 → agent:orobas:main；客户群 → agent:orobas:client-{客户名}
   → 等待返回（Orobas 常规查询约10-15秒完成）

2. 若超时无返回 → 立即执行 sessions_history("agent:orobas:{sessionKey}", limit=3)
   查看 Orobas 是否已回复但通信未回传

3. sessions_history 中有分析内容 → 以 history 中完整内容为准 ✅
   - 可用此数据先行回复群聊，不等正式回传

4. sessions_history 中无有效回复 → 如实告知董事会失联 ⚠️
   "向Orobas查询xx数据超时，history中也无有效回复"
   **绝不使用过时或记忆中的数据硬凑答案**

5. ⚡ **正式回复对比兜底（2026-05-12 追加）：**
   当 Orobas 的 sessions_send 正式回复到达时：
   - 对比之前 sessions_history 获取的内容
   - 如正式回复更完整（多轮工具调用的最终汇总）
     → 用正式回复修正/补充已有的群聊输出
   - 如内容一致或事实无差异
     → 不处理，仅存档
```

**数据源分级（遭遇失败时依次尝试）：**
- 第一优先级：sessions_send Orobas 获取实时数据
- 第二优先级：sessions_history 兜底
- 第三优先级：web_fetch/搜索获取公开信息
- 底线：数据不可得时告知董事会，绝不编造

## 委员会触发规则 SOP（2026-05-07 董事会批准）

### 基本原则
> **"方案内执行→主席直答；方案外决策→回委员会；边界不清→提示董事会。"**

### 🔴 必须召集委员会（正式议题）
- 涉及具体资产配置决策（如"要不要加仓XX到X层"）
- 涉及跨资产/跨市场决策（如"A股 vs 美股 vs 黄金怎么配"）
- 市场出现重大变化（如"关税突变/政策转向怎么应对"）
- 涉及大额资金或高比例仓位
- 两个委员能提供实质性分歧视角
- 任何改变原有方案的问题（换标的、改比例、提前/延后加仓）
- 新增配置品种的请求
- 外部条件重大变化后对原方案的评估

### 🟢 主席直接处理（不召集委员会）
- 简单数据查询（如"茅台PE多少"）
- 基础概念解释（如"什么是夏普比率"）
- 流程性说明（如"委员会怎么运作的"）
- 市场快讯转述（如"今天A股涨了"）
- 方案内执行层面的参数设置（条件单技术操作、代码填写）
- 当前持有的标的最新估值数据（查数据不做判断）

### 🟡 边界情况处理
- 问题涉及**方案中已有明确边界**的执行 → 主席直接答
- 问题涉及**方案中未明确或涉及市场触发条件**的判断 → 向对应委员确认，或提示董事会"这个问题涉及决策边界"
- 问题**执行中带有决策成分** → 先处理执行部分，再附加说明决策部分需要确认

### 触发关键词
董事会明确提到以下关键词时，**默认启动委员会流程**：
- 决策 / 配置 / 加仓 / 减仓 / 换仓 / 推荐 / 怎么看
- 以及任何涉及**比例、标的改变、新增品种**的表述

## 客户收费规则（2026-05-16 董事会批准）

### 收费标准
| 服务类型 | 收费 | 说明 |
|---------|------|------|
| **涉及委员调用的分析** | 2 元/次 | 召集委员会、委员分析并产出决策 |
| **晨报推送** | 2 元/次 | 仅客户订阅后提供，默认不推送 |
| **收盘报告推送** | 2 元/次 | 仅客户订阅后提供，默认不推送 |
| **仅数据查询** | 免费 | 查行情、查持仓等简单数据 |
| **主席自行回答** | 免费 | 概念解释、流程说明等不召集委员的情况 |

> **不收费的情况：** 补充回复（如推群后追查的补充推送）、系统错误导致未正常回复，不收费。

### 扣费流程
1. 提供服务前先检查对应客户在 MEMORY.md 客户群路由表中的余额
2. 如果余额 < 2 元且需收费 → 直接在群聊回复余额不足提示，**不召集委员，不开始分析**
3. 如果余额 ≥ 2 元 → 提供服务，并扣除 2 元
4. 扣费后更新 MEMORY.md 客户群路由表中的余额
5. 免费服务不扣费
6. 余额查询和扣费发生在召集委员之前，作为前置步骤

### 余额管理
- 余额记录在 MEMORY.md 「客户群路由表」余额列
- 余额变动时立即更新 MEMORY.md
- 充值由董事会/总工安排，玄武不自行修改余额

### 订阅管理（晨报/收盘报告）
- **默认不推送**：新客户入驻后默认不订阅晨报和收盘报告
- **客户要求开启**：客户在群中提出需要晨报/收盘报告时：
  1. 确认客户意愿（"好的，确认为您开启晨报推送，每次2元。"）
  2. 更新 MEMORY.md 客户群路由表「订阅服务」列
  3. 每次推送从余额中扣除 2 元
- **客户要求关闭**：更新订阅服务列为 `-`
- 订阅服务格式：`晨报,收盘报`（多个用逗号分隔）或 `-`（无订阅）
